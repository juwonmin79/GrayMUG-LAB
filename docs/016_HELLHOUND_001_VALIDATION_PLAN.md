# Hellhound-001 Validation Plan

## 1. Purpose

Hellhound-001 validates whether the LAB Lead Line Universe improves Hound target discovery without modifying Production Hound.

The validation compares two universes over the same observation period:

- Production Hound Universe
- LAB Lead Line Universe

This plan does not implement trading, automatic execution, position sizing, order placement, or Production Hound modification.

## 2. Core Question

Does the Lead Line Universe identify better Hound targets than the Production Hound Universe when both are observed under the same market window?

Better means:

- More useful alerts.
- Earlier first detection.
- Better forward returns.
- Better BTC-relative returns.
- Higher hit rate.

## 3. Safety Rules

- Do not modify Production Hound.
- Do not modify Hound scanner thresholds.
- Do not replace Hound RSI, volume, BTC relative strength, taker, MACD, whale, or BTC guard logic.
- Do not add execution logic.
- Do not add automatic orders.
- Do not add position management.
- Do not add Forecast, Graph ML, or Whale ML.
- Use Hellhound as a validation surface only.

## 4. Comparison Design

Both universes must be evaluated over the same period, using the same timestamp boundaries and the same market data source.

```text
Observation Window
        |
        +-- Production Hound Universe
        |
        +-- Lead Line Universe
```

The comparison should record candidates and alerts from both sides without changing either side's baseline logic.

## 5. Universe Definitions

### Production Hound Universe

The Production Hound Universe is the symbol set generated by the production scanner path.

Reference:

- `backup_GrayMUG/hound/scanner.py`
- `HoundScanner.get_top_symbols()`

Expected behavior:

- Top symbols by market volume under the configured quote asset.
- Stable assets and configured reserve assets excluded.
- BTC excluded from alt scan universe.

### Lead Line Universe

The Lead Line Universe is the LAB-provided universe from the Lead Line API Socket.

Reference:

- `research/whale_link_flow/lead_line_socket.py`
- `get_hound_universe(mode, top_n, min_priority)`

Expected behavior:

- Priority-ranked LAB target universe.
- Mode-aware output for `BEAR_ESCAPE`, `BTC_ACCUMULATION`, or `OBSERVE_ONLY`.
- No direct Hound logic replacement.

## 6. Required Metrics

### 1. Alert Count

Definition:

- Number of valid alerts generated by each universe during the observation window.

Required fields:

- `universe_type`
- `symbol`
- `alert_timestamp`
- `alert_type`
- `source`

Interpretation:

- Higher alert count is not automatically better.
- Alert count must be evaluated with hit rate and forward return.

### 2. First Detection Time

Definition:

- The earliest timestamp at which a symbol is detected by each universe.

Comparison:

- For overlapping symbols, compare which universe detected the symbol first.
- For universe-only symbols, record as exclusive first detection.

Required fields:

- `symbol`
- `production_first_detection`
- `lead_line_first_detection`
- `lead_time_delta_minutes`
- `winner`

Allowed winner values:

- `PRODUCTION_HOUND`
- `LEAD_LINE`
- `TIE`
- `PRODUCTION_ONLY`
- `LEAD_LINE_ONLY`

### 3. Forward Return

Definition:

- Price return after alert timestamp.

Required horizons:

- `+1h`
- `+4h`
- `+24h`

Formula:

```text
forward_return_h = (price_at_horizon - alert_price) / alert_price
```

Required fields:

- `symbol`
- `alert_timestamp`
- `alert_price`
- `price_1h`
- `price_4h`
- `price_24h`
- `forward_return_1h`
- `forward_return_4h`
- `forward_return_24h`

Notes:

- Missing horizon data must be marked as missing, not filled with zero.
- No look-ahead data may be used before the alert timestamp.

### 4. BTC Relative Return

Definition:

- Asset forward return minus BTC forward return over the same horizon.

Formula:

```text
btc_relative_return_h = asset_forward_return_h - btc_forward_return_h
```

Required horizons:

- `+1h`
- `+4h`
- `+24h`

Required fields:

- `symbol`
- `asset_forward_return_1h`
- `asset_forward_return_4h`
- `asset_forward_return_24h`
- `btc_forward_return_1h`
- `btc_forward_return_4h`
- `btc_forward_return_24h`
- `btc_relative_return_1h`
- `btc_relative_return_4h`
- `btc_relative_return_24h`

Interpretation:

- Positive BTC-relative return means the target outperformed BTC over that horizon.
- GrayMUG final evaluation remains BTC accumulation oriented.

### 5. Hit Rate

Definition:

- Percentage of alerts whose forward return or BTC-relative return exceeds the configured success threshold.

Initial success threshold:

```text
forward_return_4h > 0
```

Secondary threshold:

```text
btc_relative_return_4h > 0
```

Formulas:

```text
absolute_hit_rate = successful_absolute_alerts / total_evaluable_alerts
btc_relative_hit_rate = successful_btc_relative_alerts / total_evaluable_alerts
```

Notes:

- Alerts missing the `+4h` price are excluded from hit rate denominator.
- The threshold may be documented per run, but must not be changed during a run.

## 9. Validation Procedure

1. Define observation window.
2. Capture Production Hound Universe snapshots.
3. Capture Lead Line Universe snapshots.
4. Record alerts generated from each universe.
5. Record alert price at detection time.
6. Collect future prices at `+1h`, `+4h`, and `+24h`.
7. Calculate absolute forward returns.
8. Calculate BTC forward returns over the same horizons.
9. Calculate BTC-relative returns.
10. Calculate hit rates.
11. Compare Production Hound Universe vs Lead Line Universe.

## 10. Decision Criteria

Lead Line Universe is considered useful if it shows one or more of:

- Earlier first detection for shared targets.
- Higher `+4h` forward return.
- Higher `+4h` BTC-relative return.
- Higher absolute hit rate.
- Higher BTC-relative hit rate.
- Lower noise when alert count is similar or lower.

Lead Line Universe is not considered useful solely because it produces more alerts.

## 11. Forbidden Conclusions

Hellhound-001 must not conclude:

- LAB should replace Hound detection logic.
- LAB should change Hound thresholds.
- LAB should execute trades.
- LAB should set TP/SL.
- LAB should manage positions.

The only allowed conclusion is whether Lead Line Universe is a better candidate universe or priority overlay for future Hellhound experiments.
